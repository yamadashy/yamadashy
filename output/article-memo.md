## 得られた洞察

### 1. 小さな改善の積み重ね
- 1つの大きな最適化ではなく、10の小さな最適化を積み上げ
- 各コミットでベンチマーク結果を記録
- 効果がなくても「効果なし」として記録（空コミット設計）

### 2. 依存関係削減が意外と効果的
- tiktoken削除 → 41.5%改善（最大のインパクト）
- Handlebars削除 → 40.2%改善
- log-update削除 → 49ms削減
- **ライブラリを減らすことが最も効果的だった**

### 3. AIがアーキテクチャ変更を判断
- ワーカープール廃止（メインスレッド移行）
- tiktoken(WASM)→gpt-tokenizer(純JS)への置換
- **「依存ライブラリを置き換える」という人間が躊躇する判断をAIが行った**

### 4. 最適化手法の多様性
Claudeが自律的に発見・適用した手法：
- アルゴリズム改善（O(n²)→O(n)、Set/Map活用）
- 遅延読み込み（動的インポート28箇所+）
- 並列化（Promise.all、パイプライン並列化）
- 依存関係削減/置換
- 起動最適化

### 5. AIレビューとの連携
- Gemini Code Assistが自動レビュー
- 「正しく、ベストプラクティスに準拠」と承認
- レビュー指摘→修正のサイクルが自動化

### 6. ベンチマーク基盤が必須
- PRにpushするたびにCI自動ベンチマーク
- 結果がコメントに投稿される
- **この基盤がなければ「数値で判断」が成立しない**

## /schedule の汎用性

### このアプローチが有効なタスク
1. **パフォーマンスチューニング** - 数値で良し悪しが判断可能
2. **CI failure自動修正** - テストが通るかで判断可能
3. **ドキュメント更新** - 差分が明確
4. **依存関係アップデート** - テストが通るかで判断可能
5. **コードスタイル統一** - lintで判断可能

### 共通要件
- **客観的な成功基準がある**（ベンチマーク、テスト、lint）
- **非破壊的に試行できる**（テストが通れば正しい）
- **クリエイティブな判断が不要**（設計判断ではなく実装改善）

## プロンプト全文

```markdown
# 自動パフォーマンスチューニング
`src`、`website/server`、および関連コード（テスト、設定、依存関係）のパフォーマンス改善またはメモリ消費削減を行う。
アルゴリズム変更、アーキテクチャ再構成、並列化、キャッシュ戦略、ライブラリ置換、依存関係アップグレード、I/O削減、ピークメモリ削減、メモリリーク修正、起動時間短縮など、幅広く検討する。1000ファイル規模で数ミリ秒しか改善しないような軽微な変更は対象外。測定可能なインパクトのある変更を目指す。
## 1. ブランチ準備
- `perf/auto-perf-tuning` ブランチで作業する
- ブランチがなければ最新 `main` から作成、あればチェックアウトする
- `main` が先に進んでいる場合は rebase する
## 2. 既存PRの確認
- `perf/auto-perf-tuning` に対するPRがすでに存在するか確認する
- 存在する場合、PRの説明とレビューコメントを読み、コンテキストを把握する
## 3. 調査・計画
5つの重複しない調査スコープを定義する（ディレクトリ境界、横断的関心事（I/O、メモリ、並列性、アルゴリズム計算量、依存関係の重さ）、パイプラインステージなど）。5つのエージェントを並列に起動し、それぞれ1スコープを担当させる。全エージェントの報告を統合し、改善計画を立てる。
複数の改善点が見つかっても、1つのPRに収まる範囲にスコープを絞り、最もインパクトの大きい変更に集中する。
## 4. 実装
計画を実装する。
## 5. ベンチマーク・検証
- ベンチマークを実行し、変更が確実に改善であることを測定で確認する
- `npm run lint` と `npm run test` を実行し、リグレッションがないことを確認する
## 6. コミット
- Conventional Commits 形式でコミットする
- コミットボディに対応内容とベンチマーク結果を詳しく記載する
- ベンチマーク上効果がなかった場合でも、空コミットでメッセージとボディに結果を記録する
## 7. PR作成・更新
- PRがすでに存在する場合は更新、なければ新規作成する
- Draft で作成する
- タイトル: `perf(core): Automated performance tuning by Claude`
- 説明に対応内容とベンチマーク結果を追記する
## 8. レビュー対応
- ブランチへの push 後、適度に sleep しつつ AI からのレビューコメントを確認する
- レビュー指摘を検討し、必要な修正を行う
- 修正 → push → レビュー確認のサイクルを繰り返す
```

## 意思決定記録

### なぜパフォーマンスチューニングを選んだか
- 数値で良し悪しが判断できる（ベンチマークという客観指標）
- 非破壊的にできる（テストが通れば正しい）
- クリエイティブではない（設計判断ではなく実装改善）
→ AIが自律的に回せる理想的なタスク

### なぜ5並列エージェントか
- 調査スコープを分割して網羅性を確保
- 1エージェントでは見落とす可能性
- 各エージェントが異なる視点（I/O、メモリ、並列性、アルゴリズム計算量、依存関係）

### なぜ空コミットでも記録するか
- 試行錯誤の履歴が残る
- 「効果なし」も重要な情報
- 同じ最適化を重複して試さない
